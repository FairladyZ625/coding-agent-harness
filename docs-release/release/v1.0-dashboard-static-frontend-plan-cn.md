# v1.0 Dashboard Static Frontend Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 把当前 `harness dashboard` 从“CLI 结果网页化”重做成一个可读的 PLM 项目驾驶舱，用文件夹承载静态前端、表格数据、Markdown 阅读和 Mermaid 渲染。

**Architecture:** v1.0 dashboard 仍然是静态产物，不引入 React runtime。CLI 生成一个 dashboard 文件夹，里面包含 `index.html`、CSS/JS assets、规范化 JSON、Markdown 文档快照；浏览器端负责 tab、filter、drawer、Markdown render、Mermaid render、i18n 和 light/dark。这样 1.0 仍然能离线打开、能作为 evidence artifact，也为未来 React TS 迁移保留清晰组件边界。

**Tech Stack:** Node.js CLI, vanilla ES modules, CSS variables, local vendored `marked` or `markdown-it`, local vendored Mermaid, generated JSON snapshots, no backend, no React in v1.0.

## 0. 核心判断

当前 dashboard 最大问题不是样式，而是信息架构错了：它把 `status --json` 结果展示出来，却没有围绕 harness 的真实核心建模。

Harness 的核心是几张表：

| Layer | 表 | Dashboard 价值 |
| --- | --- | --- |
| 全局总账 | `docs/Harness-Ledger.md` | 项目现在有哪些任务、是否收口、证据在哪里、residual 是什么 |
| 模块/交付 | `Module-Registry.md`, `Feature-SSoT.md`, `Delivery-SSoT.md` | 模块状态、依赖、并行工作、handoff |
| 回归/证据 | `Regression-SSoT.md`, `Cadence-Ledger.md` | 哪些 gate 已验证、证据深度、风险缺口 |
| 教训 | `Lessons-SSoT.md`, lesson detail docs | 哪些教训沉淀了，哪些已经回写规范 |
| 任务小表 | `task_plan.md`, `execution_strategy.md`, `visual_roadmap.md`, `progress.md`, `review.md`, `findings.md`, `references/INDEX.md`, `artifacts/INDEX.md` | 单个任务的计划、执行策略、路线图、证据、审查、产物 |
| 收口 | walkthrough docs, `Closeout-SSoT.md` | 完成任务是否有可审计结论 |

Dashboard 不应该替代这些 Markdown 表，而应该成为这些表的**可视化索引 + 阅读器**。

## 1. v1.0 Dashboard 范围

### 1.1 做

- 生成一个 dashboard 文件夹，而不是单个 HTML。
- 首屏采用已选定的 B 风格：PLM 项目驾驶舱。
- 默认中文，支持 ZH / EN 切换。
- 支持 light / dark。
- 读取并规范化 harness 大表和任务小表。
- 支持 Markdown 渲染阅读。
- 支持 Mermaid code fence 渲染。
- 支持右侧详情 drawer：表格行点击后直接看对应 Markdown 原文/渲染结果。
- 支持旧项目 `legacy-compat`：把升级缺口归为 adoption advice，不和真实 blocker 混在一起。
- 保持 read-only，不写目标项目文件。

### 1.2 不做

- 不做在线编辑。
- 不做任务状态保存。
- 不做多人协作。
- 不做后端服务。
- 不做 React TS app。
- 不做自动迁移。
- 不把所有 Markdown 转成一个巨大页面。

## 2. 输出目录结构

命令：

```bash
node scripts/harness.mjs dashboard --out-dir tmp/harness-dashboard /path/to/project
```

生成：

```text
tmp/harness-dashboard/
├── index.html
├── assets/
│   ├── app.css
│   ├── app.js
│   ├── i18n.js
│   ├── markdown-reader.js
│   ├── mermaid-renderer.js
│   └── vendor/
│       ├── marked.min.js
│       └── mermaid.min.js
├── data/
│   ├── status.json
│   ├── tables.json
│   ├── documents.json
│   ├── graph.json
│   └── adoption.json
└── README.md
```

说明：

- `status.json`：当前 `harness status --json`。
- `tables.json`：从所有核心 Markdown 表解析出的 normalized tables。
- `documents.json`：被 dashboard 引用的 Markdown 文档快照，浏览器无需访问原项目路径。
- `graph.json`：模块、任务、依赖、handoff 的图数据。
- `adoption.json`：legacy / capability / strict check 的升级建议。

## 3. 页面信息架构

### 3.1 总览

回答四个问题：

- 项目现在安全吗？
- 任务走到哪了？
- 什么在阻塞？
- 下一步做什么？

首屏结构：

```text
Project Status Banner
├── 一句话状态
├── Release readiness
├── Active blockers
├── Evidence completeness
└── Next best action

Harness Ledger Highlights
├── Open tasks
├── In review
├── Missing evidence
└── Closed with residual

Milestone Roadmap
Risk Summary
Recent Review Activity
```

### 3.2 任务

以 task 文件夹为中心，不以 JSON 为中心。

```text
Task Table
├── Task ID
├── Title
├── State
├── Phase
├── Completion
├── Owner / Handoff
├── Evidence Depth
├── Review State
└── Next Action

Task Detail Drawer
├── Plan
├── Execution Strategy
├── Visual Roadmap
├── Progress
├── Review
├── Findings
├── References
├── Artifacts
└── Walkthrough
```

任务详情页不是只渲染整份 `task_plan.md`。它必须把独立的
`execution_strategy.md` 和 `visual_roadmap.md` 提升成一等视图：

| Tab | Source | Dashboard 表达 |
| --- | --- | --- |
| Plan | `task_plan.md` Goal / Scope / Acceptance | 摘要卡 + 原文阅读 |
| Execution Strategy | `execution_strategy.md` | 执行模式、subagent 使用、冲突控制、验证深度、handoff 规则 |
| Visual Roadmap | `visual_roadmap.md` | Mermaid 路线图 + phase table + 完成度/阻塞/证据状态 |
| Progress | `progress.md` | 时间线 + evidence links |
| Review | `review.md` | findings、confidence challenge、verifier verdict |
| Evidence | `artifacts/INDEX.md`, `references/INDEX.md`, progress evidence | 证据矩阵和引用索引 |

如果一个非平凡任务缺少 `execution_strategy.md` 或 `visual_roadmap.md`：

- 在任务页显示 `Plan Contract Missing`；
- 在旧版升级页归类为 `Task Contract Adoption Advice`；
- 在 declared/strict 模式下作为 checker failure；
- 在 legacy-compat 模式下作为 adoption warning。

### 3.3 模块与依赖

以 `Module-Registry.md` 和 module task plans 为主。

```text
Module Registry
Dependency Graph
Parallel Work Lanes
Handoff Queue
Global Sync Owner
Module Execution Strategy
Module Visual Roadmap
```

模块页也必须展示策略与路线图：

- module-level：来自模块目录的 `execution_strategy.md` 和 `visual_roadmap.md`，`module_plan.md` 只负责模块目标、步骤和当前状态。
- task-level：来自每个 module task 的 `execution_strategy.md` 和 `visual_roadmap.md`。
- graph-level：把 module roadmap 的 phase 依赖和 task roadmap 的 phase 依赖合并成视图。

### 3.4 风险、证据与评审

把 warning、risk、adoption advice 分开。

```text
Risk Board
├── Release blockers
├── P0/P1/P2/P3
├── Owner
├── Required action
└── Disposition

Evidence Matrix
├── command
├── diff
├── fixture
├── screenshot
├── review
├── verifier
└── live check

Review Loop
├── reviewers
├── confidence challenge
├── verifier template_id
├── verdict
└── residual routing
```

### 3.5 教训

Lessons 是 harness 变聪明的核心，不应该藏在 warning 里。

```text
Lessons SSoT
├── pending
├── approved
├── merged
├── rejected
└── superseded

Lesson Detail Drawer
├── Markdown render
├── source task
├── conflict statement
└── target reference update
```

### 3.6 旧版升级

只展示建议，不自动迁移。

```text
Legacy Adoption Assistant
├── current mode: legacy-compat / declared-capability
├── strict check result
├── missing capability registry
├── old review schema warnings
├── missing visual roadmap warnings
├── dry-run planned files
└── manual adoption steps
```

### 3.7 设置

v1.0 设置只做阅读体验，不做项目配置写入。

```text
Settings
├── language: zh / en
├── theme: light / dark / system
├── warning grouping
├── table density
├── markdown render / source toggle
└── data snapshot info
```

## 4. Markdown 与 Mermaid 策略

### 4.1 Markdown Reader

每个 dashboard 表格行都可以打开 drawer：

```text
Drawer Header
├── source path
├── copied snapshot time
├── open contract-file tabs
└── render/source toggle

Drawer Body
├── rendered markdown
├── rendered mermaid diagrams
└── table anchors
```

Markdown 文件不从浏览器读取原路径，而是在生成时写入 `data/documents.json`：

```json
{
  "documents": [
    {
      "id": "doc-001",
      "path": "TARGET:docs/Harness-Ledger.md",
      "title": "Harness Ledger",
      "content": "# Harness Ledger ...",
      "tables": ["table-ledger-main"]
    }
  ]
}
```

### 4.2 Mermaid

v1.0 使用 vendored Mermaid：

- 生成时扫描 Markdown code fence。
- 浏览器端 render ` ```mermaid ` blocks。
- render 失败时展示原始 Mermaid code 和错误提示。
- 不让 Mermaid 阻塞整个页面。

## 5. Data Model

### 5.1 Normalized Table

```json
{
  "id": "table-harness-ledger",
  "kind": "harness-ledger",
  "source": "TARGET:docs/Harness-Ledger.md",
  "columns": ["ID", "Task", "Review", "Evidence", "Residual", "Status"],
  "rows": [
    {
      "id": "HL-20260517-002",
      "cells": {},
      "links": {
        "taskPlan": "doc-task-plan",
        "review": "doc-review"
      },
      "derived": {
        "state": "implemented",
        "risk": "warn",
        "nextAction": "PR review"
      }
    }
  ]
}
```

### 5.2 Document Link

所有 path 都用前缀：

- `TARGET:`
- `PUBLIC:`
- `PRIVATE:`
- `URL:`
- `EXTERNAL:`

Dashboard 内部不暴露本机绝对路径。

## 7. Acceptance Criteria

Dashboard revamp is done when:

- The implementation task folder contains `execution_strategy.md` and `visual_roadmap.md` before code edits start.
- `harness dashboard --out-dir tmp/harness-dashboard <target>` generates a folder.
- `index.html` opens without dev server.
- Overview does not dump raw warnings.
- Harness Ledger is visible as a first-class table.
- Task detail drawer renders `task_plan.md`, `progress.md`, and `review.md`.
- Task detail drawer has first-class `Execution Strategy` and `Visual Roadmap` tabs sourced from standalone files.
- Module page has module-level strategy and roadmap views when the module folder provides standalone files.
- Mermaid diagrams render or fail gracefully.
- ZH/EN and light/dark work without re-generating.
- Mingjing docs target shows legacy adoption as guidance, not as unexplained failure.
- `npm test` covers example project and local Mingjing smoke when available.
- No target project files are modified by dashboard generation.

## 8. React TS Boundary

Do not start with React in v1.0.

Move to React TS only when at least two of these become true:

- dashboard needs persistent user state;
- dashboard needs editing or write-back;
- routes and drawer state become hard to manage in vanilla JS;
- component count exceeds the static shell maintainability threshold;
- live backend or watch mode becomes a v1.1/v1.2 requirement.

For now, implement vanilla components with React-compatible boundaries:

```text
DashboardShell
SidebarNav
OverviewPage
LedgerTable
TaskTable
TaskDrawer
MarkdownReader
MermaidBlock
ModuleGraph
EvidenceMatrix
LessonsPage
AdoptionAssistant
SettingsPanel
```

This keeps v1.0 small while preventing a future React migration from becoming a rewrite.
