# v1.0 Dashboard Static Frontend Execution Strategy

| Decision | Choice | Notes |
| --- | --- | --- |
| execution mode | coordinator-led implementation with adversarial review | Dashboard IA touches CLI output, parser, generated assets, and UX; coordinator owns integration to avoid fragmented UI decisions. |
| subagents | reviewer-only by default | Use 3 independent reviewers for IA/readability, parser/data contract, and release/smoke gates. Do not split UI implementation across workers until the static shell boundary is stable. |
| conflict control | coordinator owns shared CLI and dashboard shell | No worker writes to `scripts/harness.mjs`, `scripts/lib/harness-core.mjs`, or dashboard shell files in parallel. |
| implementation order | data contract first, UI shell second, reader/renderers third, polish last | Prevents visual work from outrunning table/data semantics. |
| verification depth | L2 for parser/UI smoke; L3 for Mingjing legacy adoption smoke | Must prove generated dashboard works for `examples/minimal-project` and `/Users/lizeyu/Projects/mingjing-app/docs` without mutating target docs. |
| React boundary | no React in v1.0 | Implement vanilla static SPA with React-compatible component boundaries for future migration. |

Execution rule: every implementation task created from this plan must refine this
strategy into its own task-folder `execution_strategy.md`. If the task uses
subagents, the allowed input packet, write scope, and handoff format must be
recorded before code edits.
